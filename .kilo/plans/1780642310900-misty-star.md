# Plan: Fix stale com-bridge (8765) after git pull for Windows desktop users (CORS on X-GCS-Tab-Id)

## 1. Problem Summary
- **Symptom**: After `git pull`, users who installed via `windows/GCS-智能安装.bat` (desktop + Startup shortcuts) see repeated errors when launching via desktop icon:
  ```
  Access to fetch at 'http://127.0.0.1:8765/bridge-open' from origin 'http://127.0.0.1:8766'
  blocked by CORS policy: Request header field x-gcs-tab-id is not allowed by Access-Control-Allow-Headers in preflight response.
  ```
  Followed by `net::ERR_FAILED` on bridge-open/close, and auto-connect loops failing.
- **Affected users**: Packaged desktop installs (8766 runtime + 8765 bridge long-lived pythonw processes). Dev machines using Live Server + direct launches are usually unaffected.
- **Root cause** (confirmed via code inspection):
  - The multi-tab bridge sharing feature (added to support concurrent tabs without double serial open) requires a custom header `X-GCS-Tab-Id` on all `/bridge-*` calls (see `JS/services/serial.js:40,45,826,840` and `JS/services/com-bridge.js`).
  - Browser therefore sends CORS preflight (OPTIONS) to 8765.
  - `tools/com-bridge/server.py:1315` (current code) correctly advertises `Access-Control-Allow-Headers: Content-Type, X-GCS-Tab-Id` in `do_OPTIONS`.
  - However, all launcher services (gcs-watchdog, gcs-runtime, etc.) only ever advertise `Content-Type`.
  - **The actual running process on 8765 is an old in-memory version of server.py** (from install time or previous run). `git pull` only updates files on disk; the background `pythonw.exe` (started via Startup "GCS Watchdog.lnk" or prior GCS.cmd) keeps the old `Handler` class loaded.
  - Windows launch robustness is missing:
    - `tools/gcs_ports.py:10`: `if sys.platform == "win32": return` (reap is a no-op; only Unix lsof works).
    - `tools/gcs_supervisor.py:132`: `if not force_restart and bridge_healthy(): return True` — reuses whatever is listening on 8765.
    - `windows/GCS.cmd:10-14`: If 8766 `/ping` succeeds, it just opens the browser page (no restart of anything).
    - `windows/GCS-智能安装.bat`, `setup-python-deps.ps1`, `install-gcs-desktop.ps1`: Only manage .venv + shortcuts (including Startup prewarm). Never kill processes.
    - `gcs-stop.py` and `gcs_watchdog.py`/`gcs-runtime.py` only reap/stop on "not healthy".
  - Result: post-pull launches (or auto-prewarm) keep serving the old bridge that doesn't know about the `X-GCS-Tab-Id` header → preflight fails.
- Why dev desktop "just works": frequent process restarts during development hit the latest `server.py` on disk.

## 2. Why "让用户重新安装" (re-run the .bat) is NOT sufficient today
- Re-running `GCS-智能安装.bat` will:
  - Re-validate/fix official Python.
  - Possibly recreate `.venv`.
  - Re-create desktop/start menu/Startup shortcuts.
- It does **not** kill running pythonw listeners on 8765/8766/8767/8768.
- Old processes survive → same stale code problem persists until manual Task Manager kill + reboot.
- User would still need extra manual steps (violates "不要让用户操作").
- Future `git pull` would regress the same way.

## 3. Goals for the Fix
- Make "re-run the smart installer" a **true one-click recovery** for affected users (after our changes land).
- Make the system **self-healing** on normal desktop launches after `git pull` (no reinstall required in most cases).
- Preserve fast "click icon → just bring existing UI to front" UX when services are already healthy and up-to-date.
- Minimal user-visible disruption (bridge restart is acceptable; JS already has reconnect/ensure logic).
- Cover the bridge (8765) primarily; apply similar patterns to other services (8768 tile etc.) for future-proofing.
- No new runtime dependencies (use stdlib + built-in Windows tools like PowerShell/netstat/taskkill).

## 4. High-Level Solution
Two complementary mechanisms:

**A. Server-side stale-code detection + auto force-restart (self-healing)**
- Add `scriptMtime` (and optionally `scriptPath`) to the bridge `/health` (and `/bridge-status`) responses.
- In `ensure_bridge_process` (and symmetric for tile server), before early-returning on "healthy", query the live listener's `/health`, compare its reported `scriptMtime` vs. `os.path.getmtime(SERVER_SCRIPT)` on disk.
- If disk mtime is newer (within small tolerance for FS clock skew), treat as stale → force `_stop_locked()` + reap + spawn fresh process with latest code.
- This triggers automatically from all call sites:
  - `gcs-runtime.py` bridge_warmup_loop + `/__gcs/ensure-bridge` (called by JS on page load / ensureComBridgeRunning).
  - `gcs_watchdog.py` launch paths.
  - `gcs_supervisor` internal watchdog_loop.
- No change to JS required for the core fix (existing `requestBridgeStartup` + ensure flows will cause the Python side to do the right thing). Optional: enhance `probeBridgeHealth` to surface the mtime for diagnostics/logging.

**B. Aggressive cleanup in the install path (makes reinstall reliable)**
- Create a pure-PowerShell early stopper (`windows/tools/stop-gcs-services.ps1`) that kills listeners on the 4 GCS ports (8765-8768) if owned by any python* process. Runs with zero Python/.venv dependency.
- Update `GCS-智能安装.bat` to call this PS1 (and later `gcs-stop.py --force`) at the very start, before any Python check or venv work. This guarantees a clean slate for the new install.
- Make `gcs-stop.py` support `--force` (always reap all ports, even if runtime looks healthy).
- Optionally, call the PS1 early in `windows/GCS.cmd` only in the "we are about to (re)start services" branch (to avoid killing a just-started good instance on normal reuse clicks). With mechanism A above, this is largely unnecessary for bridge, but helps for full cleanup on launch after pull.

**Result for users**:
- One-time: after the fix is in the repo, affected users double-click `windows/GCS-智能安装.bat` again → it stops everything old → fresh .venv + latest code + new shortcuts.
- Ongoing: normal desktop launches after future `git pull` will auto-refresh the bridge (via mtime check) when the page/ensure logic runs. Reinstall only needed for bigger env changes.

## 5. Detailed File Changes (in priority order)

### 5.1 New file: `windows/tools/stop-gcs-services.ps1`
- Pure PowerShell, no deps.
- Idempotent, silent success, logs only on error (append to GCS_INSTALL_LOG if present).
- Logic:
  ```powershell
  $ports = @(8765, 8766, 8767, 8768)
  foreach ($p in $ports) {
    Get-NetTCPConnection -LocalPort $p -State Listen -ErrorAction SilentlyContinue |
      Select-Object -ExpandProperty OwningProcess -Unique |
      ForEach-Object {
        $proc = Get-Process -Id $_ -ErrorAction SilentlyContinue
        if ($proc -and ($proc.Name -like '*python*')) {
          Stop-Process -Id $_ -Force -ErrorAction SilentlyContinue
        }
      }
  }
  ```
- Also handle older Windows without Get-NetTCPConnection (fallback to netstat parsing + taskkill).
- Called from .bat early (and optionally .cmd).

### 5.9 `windows/GCS.cmd` (and optionally GCS-Prewarm.cmd)

  - **Caveat / decision point**: Doing this unconditionally would kill a healthy running instance on every icon click, causing a visible restart. 
  - Recommended: Only do the full stop ps1 if the 8766 ping **failed** (i.e., we are in the branch that will start watchdog anyway). Or skip the ps1 here entirely and rely on the mtime-based force inside the supervisor (which only kills the bridge, not the runtime, and only when actually stale).
  - Per user decision: use option A (surgical). Rely purely on the Python-side mtime check inside ensure_bridge_process. No early explicit reaping of the bridge in the "runtime healthy, just open browser" path of GCS.cmd. The refresh will happen on-demand when the page/JS calls ensure-bridge (which goes through the supervisor). This keeps normal icon clicks fast and non-disruptive.

## 7. Tradeoffs & Risks
- **Bridge restart on stale detection** (only when mtime check decides disk is newer): Brief loss of MAVLink stream for any open tab. Mitigated by:
  - Existing subscriber model (last tab wins).
  - JS reconnect logic, heartbeats, `waitForFirstMavlinkPacket`, auto-connect.
  - Only happens when code actually changed on disk.
- Early PS1 stop in install bat: Harmless (idempotent). May briefly kill services during a long install, but install already takes time and shows progress.
- PowerShell reaping: Relies on Get-NetTCPConnection (Win8+ / Server 2012+). Fallback to netstat parsing covers older. Project already shells to PowerShell heavily (port enumeration, etc.), so acceptable.
- No change to the actual CORS headers in runtime/watchdog (they don't need the tab-id header for their own endpoints).
- If a user has a very old install (pre any of the current supervisor logic), the first re-run of the bat will clean it up.
- Clock skew on mtime comparison: Use 2-5s tolerance (per user choice) + only trigger force when disk > live (never the other way). When a mtime-based force-restart is decided, emit a one-line log to the bridge stderr log (e.g. "[stale-bridge] disk mtime X > live Y, forcing restart").
